# SeekDesk 双 Runtime 支持规格

## 目标

SeekDesk 同时支持 `cloud_runtime` 与 `local_daemon` 两种代码执行环境，并通过统一的工作区、会话、工具调用、审批和审计体验向用户提供服务。

- `cloud_runtime`：由 SeekDesk 服务端为用户创建隔离的云端工作区，从 Git 仓库导入代码，在云端执行文件读取、文件修改、搜索、Diff、Shell 和测试任务。
- `local_daemon`：由用户在自己的电脑上启动 daemon，授权 SeekDesk 操作指定的本地文件夹，在本机执行相同类型的工具任务。
- 用户在新建对话时选择工作区和 Runtime 类型。会话创建后固定绑定该工作区，所有文件、命令、工具、审批、产物和审计记录必须使用同一工作区。

## 用户

- 希望无需安装本地程序，直接从 Git 仓库开始编码的线上用户。
- 希望操作本机未推送代码、私有文件或本地开发环境的用户。
- 需要在不同设备上继续云端开发任务的用户。
- 需要明确控制文件修改、Shell 和测试执行权限的开发者或团队成员。

## 用户场景

### 创建云端工作区

1. 用户点击“新对话”。
2. 用户选择“云端工作区”。
3. 用户选择已有云端工作区，或提供一个有权访问的 Git 仓库和目标分支创建工作区。
4. SeekDesk 显示创建、克隆和初始化状态。
5. 工作区可用后创建会话，并进入对话页面。
6. Agent 的文件、搜索、Git、Shell 和测试工具在该隔离环境中运行。

### 创建本机工作区

1. 用户在本机启动并配对 `seekdesk-daemon`。
2. 用户点击“新对话”，选择在线 daemon 暴露的本地工作区。
3. 用户可通过系统目录选择器或受限的目录浏览功能更换 daemon 授权的根目录。
4. 会话绑定本地工作区，工具请求通过 API 转发到对应 daemon 执行。
5. daemon 离线时，已有会话和历史记录仍可查看，但任何依赖 Runtime 的操作不可执行。

### 在两种 Runtime 间使用一致的工具

1. 用户向 Agent 提交编程任务。
2. 只读工具可以在当前工作区自动执行。
3. 文件写入、文件编辑、Shell 和测试执行生成待审批工具调用。
4. 用户查看工具输入、目标工作区和风险说明后批准或拒绝。
5. 批准后由当前会话绑定的 Runtime 执行，不得自动切换到另一 Runtime。
6. 执行结果同步到对话、运行详情、Diff、终端、活动记录和会话 trace。

### 恢复已有会话

1. 左侧历史记录按工作区分组，并标识工作区类型和在线状态。
2. 用户打开历史会话时，SeekDesk 恢复其原始 `workspaceId` 和 Runtime 绑定。
3. 云端工作区已停止时，用户可重新启动并继续使用。
4. 本机 daemon 离线时，页面显示重新连接指引，不回退到云端或服务端本地目录。

## 范围内

- 统一工作区模型，明确区分 `cloud_runtime` 和 `local_daemon`。
- 新建对话时选择 Runtime 和工作区。
- 创建、列出、启动、停止和查看云端工作区状态。
- 从 Git 仓库和指定分支创建云端工作区。
- 注册、配对、心跳检测和展示本机 daemon 工作区。
- 会话与 `workspaceId`、Runtime 类型固定关联。
- 文件列表、文件读取、代码搜索、Git status 和 Git diff。
- 文件写入、文件编辑、Shell 和测试的会话内审批与执行。
- Runtime 请求状态：排队、初始化、运行、完成、失败、超时和取消。
- 运行详情、终端输出、Diff、活动事件、产物、模型用量和 trace 的工作区关联。
- 云端工作区资源限制、空闲回收、数据清理状态和错误提示。
- 本机 daemon 离线、云端 Runtime 不可用、Git 克隆失败等降级体验。
- 服务端保存工作区元数据、会话关联和审计数据；敏感凭据不返回浏览器。

## 范围外

- 直接从浏览器访问或修改任意本地文件夹。
- 在同一个会话中途切换工作区或 Runtime。
- 未经用户审批执行文件写入、Shell、测试或其他高风险操作。
- 自动执行 Git commit、push、force push、rebase、reset 等 Git 写操作。
- 完整 IDE 插件、多人实时光标协作和代码结对编辑。
- 云端 GPU 任务、长期常驻服务和生产环境部署。
- Runtime 之间自动同步未提交文件。
- 第一版中的企业级组织策略、细粒度角色权限和自带云账号部署。

## 验收标准

### Runtime 与工作区选择

- 新建对话入口同时展示可用的云端工作区和在线本机 daemon 工作区。
- 每个工作区清晰显示名称、Runtime 类型、根目录或仓库、分支、连接状态和最近使用时间。
- 用户未选择工作区时不能创建 coding 会话，并收到明确提示。
- 会话创建后保存唯一 `workspaceId` 和 Runtime 类型；刷新页面或恢复会话后绑定保持不变。
- 历史会话按工作区分组，工作区组和会话排序稳定。

### Cloud Runtime

- 用户可以从有权访问的 Git 仓库和分支创建云端工作区。
- 创建过程中可见克隆和初始化状态，失败时返回可理解且可重试的错误。
- 不同用户、工作区和并发任务之间文件系统、进程、环境变量及网络权限相互隔离。
- Runtime 停止后不会继续执行工具请求；重新启动后能恢复持久化的工作区文件。
- 超过配置的空闲时间后 Runtime 自动停止，并在界面显示已停止状态。
- 用户删除云端工作区前必须确认；删除后代码副本和 Runtime 数据按产品的数据保留规则清理。

### Local Daemon

- daemon 通过有效配对凭据注册，并定期发送心跳。
- API 能按 `workspaceId` 将工具请求路由到唯一的在线 daemon。
- daemon 只能访问用户授权的工作区根目录，路径穿越、危险目录、大文件和二进制限制继续生效。
- daemon 断线后工作区状态及时变为离线，工具请求返回 `runtime_unavailable`，不会转发到其他 daemon 或云端 Runtime。
- daemon 重连并恢复同一路径时，工作区身份保持稳定，已有会话可继续运行。

### 工具与审批

- 两种 Runtime 对外提供一致的工具名称、输入校验、结果结构和错误结构。
- `coding.list_files`、`coding.read_file`、`coding.grep`、`coding.git_status` 和 `coding.git_diff` 可按安全策略自动执行。
- `coding.write_file`、`coding.edit_file`、`coding.run_shell` 和 `coding.run_tests` 在无有效会话授权时只能进入 pending 状态。
- 审批界面必须显示会话、工作区、Runtime、工具名、主要输入和风险说明。
- 批准后再次校验会话、工作区、授权状态和输入 schema，再交给绑定 Runtime 执行。
- 拒绝、撤销、过期或跨会话授权不能触发执行。
- 所有执行状态和结果写入 tool call、activity event 和 session trace；产生文件变更时可查看对应 Diff。

### 页面体验

- 默认对话页面保持简洁，Runtime 详情只在新建对话、工作区状态或运行详情中按需展示。
- 文件、搜索、Diff、终端和运行详情始终显示当前工作区名称和 Runtime 类型。
- Runtime 初始化、离线、停止、忙碌、超时和失败都有明确状态，不出现空白面板、问号占位或无响应按钮。
- 页面不得因为 Runtime 类型不同而出现不同含义的重复入口。
- 浏览器不得直接接收 Git 凭据、daemon 配对密钥、云端 Runtime 密钥或服务器环境变量。

### 安全与审计

- 云端 Runtime 默认使用最小权限、受限资源和受控网络访问。
- 本机 daemon 与云端 Runtime 均执行 workspace root 边界检查和命令安全策略。
- Shell 和测试具有超时、输出截断、cwd 锁定和敏感环境变量脱敏。
- 每次工具调用记录 `sessionId`、`workspaceId`、Runtime 类型、审批人或审批来源、开始时间、结束时间和执行结果。
- Runtime 不可用时不得伪造执行成功，也不得静默切换到 `server_local` fallback。

## 边界情况

- Git 仓库不存在、无访问权限、默认分支不存在或仓库过大。
- 云端 Runtime 创建成功但 Git 初始化失败。
- 云端 Runtime 启动超时、资源不足、进程异常退出或被空闲回收。
- 用户删除仍有运行中工具调用的云端工作区。
- 同一 Git 仓库创建多个云端工作区或绑定多个会话。
- daemon 在工具执行期间断线，随后使用相同或不同路径重连。
- 本机工作区被移动、删除、重命名，或 Git 状态在会话外发生变化。
- 用户在审批后、执行前撤销授权。
- 会话请求携带的 `workspaceId` 与 tool call、grant 或 Runtime 不一致。
- 多个浏览器标签同时操作同一工作区和会话。
- 云端工作区已停止但用户尝试读取文件或执行 pending 工具调用。
- Runtime 返回超大输出、非法结果、未知工具或过期请求。
- Git 凭据过期，或私有仓库访问授权被撤销。

## 假设

- 第一版云端工作区以 Git 仓库作为代码导入来源，不支持直接上传完整本地文件夹作为持续同步源。
- 云端工作区文件在 Runtime 停止后仍可恢复；删除工作区才触发数据清理。
- 一个 coding 会话只绑定一个工作区，跨 Runtime 迁移通过新建工作区和新会话完成。
- 云端工作区与本机 daemon 使用同一套 coding tool schema 和审批规则。
- 云端 Runtime 的具体容器平台或云厂商不属于产品规格，由后续技术方案决定。
- Git 凭据由服务端安全保存或通过短期授权获得，浏览器只接收授权状态。
- `server_local` 仅保留为开发和自动化测试 fallback，不作为普通用户可选 Runtime。
- 第一版不开放 Git 写操作；用户可查看 status 和 diff，提交与推送能力在后续规格中单独定义。
- 多人使用指不同用户可以拥有独立云端工作区；第一版不要求多人同时编辑同一工作区。
